## 1. 프로젝트 개요
---
### 한 줄 정의
AI 기반 자동 GitHub PR 코드 리뷰 서비스

### 제품 목표
개발팀 및 개발자가 코드 품질 및 일관성을 향상시키고, 개발 생산성과 효율성을 높이며, 코드 리뷰에 소요되는 시간을 단축하고, 잠재적인 버그 발생 가능성을 줄이는 것을 목표로 합니다.

### 배경
최근 AI 기술의 발전으로 코드 분석 및 생성 능력이 크게 향상됨에 따라, 반복적이고 시간 소모적인 코드 리뷰 프로세스를 자동화하여 개발자의 부담을 줄이고 더 높은 가치 창출에 집중할 수 있도록 돕는 서비스의 필요성이 커지고 있습니다. CodeLens는 이러한 시장의 요구에 부응하며, 1인 개발 포트폴리오 프로젝트로서 백엔드 및 프론트엔드 스택 학습을 목표로 개발되고 있습니다.

### 사용자 문제
코드 리뷰어의 부담 가중, 리팩토링 필요성 식별의 어려움, 코드 리뷰 품질의 일관성 부족, 적시에 피드백을 받기 어려움 등으로 인해 개발 과정의 비효율성과 코드 품질 저하 문제가 발생하고 있습니다.

### 해결 방안
CodeLens는 GitHub PR이 생성되면 Anthropic Claude AI를 활용하여 자동으로 코드 품질, 잠재적 버그, 리팩토링 포인트를 분석하고, 분석 결과를 GitHub PR 코멘트로 즉시 제공함으로써 개발자의 시간과 노력을 절약하고 코드 품질을 향상시킵니다.

### 차별점
CodeLens는 심각도 분류(Critical / Warning / Info)를 통해 이슈의 우선순위를 명확히 제시하며, 파일별 요약과 함께 실제 개선 코드를 스니펫 형태로 제안하여 개발자가 즉시 적용할 수 있도록 돕습니다. GitHub에 직접 통합되어 사용자 경험을 해치지 않으면서도, AI의 깊이 있는 분석과 실행 가능한 피드백을 제공하는 것이 주요 차별점입니다.

### 타겟 사용자
주요 타겟 사용자는 개인 개발자, 오픈소스 프로젝트 기여자, 그리고 소규모/중규모 개발팀에 속한 개발자입니다. 특히, '김민준' 개발자 페르소나는 사이드 프로젝트와 오픈소스에 적극적으로 참여하지만, 코드 리뷰에 많은 시간을 할애하기 어렵고, 동료 개발자의 부재로 심도 있는 코드 피드백을 받기 어려운 상황에 있는 주니어/미들 개발자입니다. 그는 자신의 코드를 개선하고 싶지만, 시간적 제약과 전문성 부족으로 어려움을 겪고 있습니다.

### 사용자 시나리오
김민준 개발자가 자신의 GitHub 저장소에 새로운 기능을 구현한 후 Pull Request를 생성합니다. PR이 생성되자마자 CodeLens는 GitHub Webhook을 통해 이를 감지하고, 해당 PR의 변경사항(diff)을 AI로 분석합니다. 잠시 후, 김민준의 PR 페이지에 CodeLens가 Critical, Warning, Info 등으로 분류된 코드 품질 피드백과 함께 각 이슈에 대한 설명, 그리고 개선된 코드 스니펫이 담긴 코멘트를 자동으로 달아줍니다. 김민준은 CodeLens의 피드백을 바탕으로 코드의 잠재적 문제를 빠르게 파악하고 개선하여, 더 높은 품질의 코드를 병합할 수 있게 됩니다.

### 핵심 지표
월별 AI 코드 리뷰 처리 PR 수 (MAU 기준), PR당 평균 피드백 코멘트 수, 사용자의 리뷰 내용 만족도 (설문조사), 신규 가입자 수, 재방문율, 피드백 반영률 (향후 측정 가능 시).

### 리스크
AI 모델의 할루시네이션(환각)으로 인한 잘못된 피드백 발생 가능성, GitHub API 사용량 제한 문제, 경쟁 서비스의 출현 및 차별성 유지의 어려움, 사용자 코드 보안 및 개인 정보 보호 문제, AI 분석 비용 효율성 유지.

## 1. GitHub OAuth 인증 및 저장소 연동
> **중요도**: 🔴 높음 | **진행 상태**: 🔵 부분 완료 (1.1 로그인은 완료, 1.2 저장소 연동은 미착수)

사용자가 GitHub 계정을 통해 CodeLens에 안전하게 로그인하고, 코드 리뷰를 받고자 하는 GitHub 저장소를 CodeLens와 연동할 수 있는 기능입니다.

**수용 기준:**
```
1. ● 사용자는 GitHub OAuth를 통해 CodeLens에 로그인할 수 있어야 합니다.
2. ○ 로그인 후, 사용자는 CodeLens가 접근할 GitHub 저장소 목록을 선택하고 연동할 수 있어야 합니다.
3. ○ CodeLens는 연동된 저장소의 PR 이벤트에 접근할 권한을 가져야 합니다.
```


### 1.1 GitHub OAuth 로그인
> **중요도**: 🔴 높음 | **진행 상태**: 🟢 완료 | **역할**: User | **기기**: Web

사용자가 GitHub OAuth 프로토콜을 통해 CodeLens에 안전하게 로그인할 수 있는 기능입니다. GitHub 계정 인증을 완료하면 CodeLens 세션이 생성됩니다.


#### 1.1.1 GitHub OAuth 2.0 인증 흐름 구현

사용자가 '로그인' 버튼을 클릭하면 GitHub OAuth 인증 엔드포인트로 리다이렉트되고, GitHub에서 사용자 인증을 완료한 후 인증 코드를 획득하여 CodeLens 서버에 전달하는 OAuth 2.0 표준 흐름을 구현합니다.
> **중요도**: 🔴 높음 | **진행 상태**: 🟢 완료 | **역할**: User | **기기**: Web

> `/auth/github` → GitHub 로그인 리다이렉트, `/auth/callback` → code 교환까지 구현 및 실제 테스트 완료.

#### 1.1.2 GitHub 액세스 토큰 발급 및 저장

인증 코드를 GitHub API에 전달하여 액세스 토큰을 발급받고, 사용자 세션과 함께 암호화된 형태로 데이터베이스에 안전하게 저장합니다.
> **중요도**: 🔴 높음 | **진행 상태**: 🟢 완료 | **역할**: User | **기기**: Web

> 명세와 다른 부분: 액세스 토큰을 DB에 암호화 저장하는 대신, JWT payload 안에 담아 HttpOnly 쿠키로 전달하는 방식으로 구현했습니다 (CLAUDE.md 보안 설계 결정 — "클라이언트에 직접 노출 안 됨"). DB에는 토큰을 저장하지 않습니다.

#### 1.1.3 사용자 정보 조회 및 세션 생성

GitHub API를 통해 인증된 사용자의 기본 정보(GitHub ID, 사용자명, 이메일 등)를 조회하고, CodeLens 데이터베이스에 사용자를 등록한 후 세션을 생성합니다.
> **중요도**: 🔴 높음 | **진행 상태**: 🟢 완료 | **역할**: User | **기기**: Web

> `users` 테이블에 github_id/login/email 저장. "세션"은 서버 세션이 아니라 JWT + HttpOnly 쿠키 방식으로 구현했습니다.

#### 1.1.4 로그인 성공 후 리다이렉트

세션 생성 완료 후 사용자를 대시보드 메인 페이지로 리다이렉트하여 CodeLens의 주요 기능을 사용할 수 있는 상태로 만듭니다.
> **중요도**: 🔴 높음 | **진행 상태**: 🔵 부분 완료 | **역할**: User | **기기**: Web

> 현재는 대시보드가 아직 없어 프론트엔드 루트(`FRONTEND_URL`)로 리다이렉트합니다. 모듈 9(대시보드)에서 실제 대시보드 경로로 변경 필요.


### 1.2 GitHub 저장소 연동 및 권한 설정
> **중요도**: 🔴 높음 | **진행 상태**: ⚪ 작성중 | **역할**: User | **기기**: Web

사용자가 로그인 후 CodeLens와 연동할 GitHub 저장소 목록을 선택하고, CodeLens의 PR 이벤트 접근 권한을 부여할 수 있는 기능입니다.


#### 1.2.1 연동 가능한 GitHub 저장소 목록 조회

GitHub API를 통해 로그인된 사용자가 권한을 가진 모든 저장소(소유 저장소, 협업 저장소 등)를 조회하여 화면에 표시합니다.
> **중요도**: 🔴 높음 | **진행 상태**: ⚪ 작성중 | **역할**: User | **기기**: Web


#### 1.2.2 저장소 선택 및 권한 부여 확인

사용자가 CodeLens와 연동할 저장소를 체크박스 또는 토글 방식으로 선택하고, CodeLens가 요청하는 권한(PR 읽기, 코멘트 작성 등)을 확인하는 UI를 제공합니다.
> **중요도**: 🔴 높음 | **진행 상태**: ⚪ 작성중 | **역할**: User | **기기**: Web


#### 1.2.3 선택된 저장소 데이터베이스 저장

사용자가 확인한 저장소 목록을 데이터베이스에 저장하여 향후 Webhook 수신 및 PR 분석 시 해당 저장소들에 대해서만 처리하도록 제한합니다.
> **중요도**: 🔴 높음 | **진행 상태**: ⚪ 작성중 | **역할**: User | **기기**: Web


#### 1.2.4 저장소별 Webhook 등록 권한 설정

선택된 저장소에 대해 GitHub API를 통해 CodeLens 서버의 Webhook URL을 GitHub 저장소에 등록할 수 있는 권한이 부여되어 있음을 확인합니다.
> **중요도**: 🔴 높음 | **진행 상태**: ⚪ 작성중 | **역할**: User | **기기**: Web

## 2. GitHub PR 이벤트 Webhook 수신 및 검증
> **중요도**: 🔴 높음 | **진행 상태**: 🔵 부분 완료 (2.1/2.2/2.3 핵심 흐름 완료, 페이로드 저장·JSON 검증·우선순위·중복방지는 미착수)

GitHub 저장소에서 PR이 생성되거나 업데이트될 때 발생하는 Webhook 이벤트를 CodeLens가 정확하게 수신하고, 이벤트의 유효성을 검증하여 보안을 확보하는 기능입니다.

**수용 기준:**
```
1. ● 연동된 저장소에서 PR 관련 이벤트(생성, 업데이트 등) 발생 시 CodeLens 서버로 Webhook이 전송되어야 합니다.
2. ● CodeLens는 수신된 Webhook 페이로드의 서명을 검증하여 신뢰할 수 있는 요청인지 확인할 수 있어야 합니다.
3. ● 유효한 Webhook 이벤트는 비동기 코드 리뷰 처리를 위한 큐에 추가되어야 합니다.
```


### 2.1 GitHub Webhook 등록 및 수신
> **중요도**: 🔴 높음 | **진행 상태**: 🔵 부분 완료 | **역할**: User | **기기**: Web

연동된 GitHub 저장소에서 PR 생성 및 업데이트 이벤트가 발생할 때, GitHub이 CodeLens 서버로 Webhook을 전송하고 서버가 이를 정확하게 수신하는 기능입니다.


#### 2.1.1 GitHub Webhook URL 등록

사용자가 저장소를 선택하면, GitHub API를 통해 CodeLens의 Webhook 수신 엔드포인트 URL을 해당 저장소에 등록하여 PR 이벤트 발생 시 GitHub이 CodeLens로 알림을 보내도록 설정합니다.
> **중요도**: 🔴 높음 | **진행 상태**: 🔵 부분 완료 | **역할**: User | **기기**: Web

> 현재는 GitHub 웹 UI에서 수동으로 한 번 등록(+ngrok 터널)해서 테스트했습니다. 저장소 선택 시 API로 자동 등록하는 기능은 아직 없고, 1.2(저장소 연동)와 함께 구현해야 합니다.

#### 2.1.2 PR 생성 이벤트 수신

PR이 생성될 때 GitHub에서 보내는 Webhook 이벤트를 CodeLens 서버의 엔드포인트에서 정확하게 수신합니다.
> **중요도**: 🔴 높음 | **진행 상태**: 🟢 완료 | **역할**: User | **기기**: Web

> 실제 GitHub PR(`opened`)로 end-to-end 테스트 완료.

#### 2.1.3 PR 업데이트 이벤트 수신

PR이 업데이트될 때(커밋 추가, 제목 변경 등) GitHub에서 보내는 Webhook 이벤트를 수신합니다.
> **중요도**: 🔴 높음 | **진행 상태**: 🟢 완료 | **역할**: User | **기기**: Web

> `synchronize` action으로 실제 테스트 완료. 제목 변경 등 다른 종류의 업데이트(`edited` 등)는 별도 테스트 안 함.


#### 2.1.4 Webhook 페이로드 저장 및 추적

수신된 Webhook 페이로드를 임시 저장소에 보관하여 나중에 서명 검증 및 이벤트 처리 시 참조할 수 있도록 합니다.
> **중요도**: 🟡 중간 | **진행 상태**: ⚪ 작성중 | **역할**: User | **기기**: Web


### 2.2 Webhook 페이로드 서명 검증
> **중요도**: 🔴 높음 | **진행 상태**: 🔵 부분 완료 | **역할**: User | **기기**: Web

수신된 Webhook 페이로드의 HMAC 서명을 검증하여 요청이 GitHub에서 발송된 신뢰할 수 있는 요청인지 확인하는 보안 기능입니다.


#### 2.2.1 GitHub Webhook 서명 HMAC 검증

GitHub의 Webhook 페이로드에 포함된 X-Hub-Signature-256 헤더의 HMAC 서명을 CodeLens의 Webhook 시크릿으로 검증하여 요청이 GitHub에서 발송되었음을 확인합니다.
> **중요도**: 🔴 높음 | **진행 상태**: 🟢 완료 | **역할**: User | **기기**: Web

> `hmac.compare_digest`로 타이밍 공격까지 방지. 정상/위조 서명 양쪽 다 테스트 완료.

#### 2.2.2 서명 검증 실패 시 처리

HMAC 서명 검증이 실패한 요청은 즉시 거부하고, 의심 로그를 기록하여 보안 위협을 모니터링할 수 있도록 합니다.
> **중요도**: 🔴 높음 | **진행 상태**: 🔵 부분 완료 | **역할**: User | **기기**: Web

> 즉시 401 거부는 구현됨. "의심 로그" 별도 기록은 아직 없고, uvicorn 기본 액세스 로그만 남는 상태입니다.


#### 2.2.3 Webhook 페이로드 JSON 파싱 검증

Webhook 페이로드의 JSON 형식 유효성을 검증하여 구조화된 데이터를 안전하게 파싱합니다.
> **중요도**: 🟡 중간 | **진행 상태**: ⚪ 작성중 | **역할**: User | **기기**: Web
